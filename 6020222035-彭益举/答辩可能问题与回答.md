## 二、通用开场问题

### 第 4-5 页：你这个课题主要解决什么问题？

推荐回答：本课题面向高校选课高峰期的并发压力、课程筛选效率低和课程时间冲突发现滞后三个问题进行设计。系统采用 Spring Cloud Alibaba 微服务架构，将基础业务和选课业务解耦，使用 Redis/Redisson 和 RabbitMQ 提升并发处理能力，同时加入基于专业匹配的课程推荐和时间窗口冲突检测，提升学生选课效率与系统稳定性。

追问应对：如果老师问“和普通选课系统区别在哪里”，重点说三点：不是单纯 CRUD，而是加入微服务拆分、高并发防护、智能推荐和冲突预警。

来源：PPT 第 4-5 页；论文 `gen2_part2.py:13-16`；论文总结 `gen2_part3.py:301-303`。

### 第 4 页：为什么传统单体架构不适合选课高峰？

推荐回答：选课高峰具有短时间集中访问特征，学生会同时登录、查询课程和提交选课请求。单体系统把登录、课程、选课、消息等模块部署在同一个应用中，一个模块性能下降会影响整体服务。微服务可以把高并发压力最大的选课服务独立出来，便于单独扩容、单独优化和故障隔离。

追问应对：如果老师问“微服务是不是一定更好”，回答不是。微服务会增加部署和服务治理复杂度，但本系统的选课服务有明显高并发和独立扩展需求，因此拆分有实际价值。

来源：PPT 第 4、7、8 页；论文 `gen2_part2.py:14-16`、`gen2_part2b.py:28`。

### 第 5 页：你的研究目标是否已经实现？

推荐回答：实现了四类目标。第一，完成基于 Spring Cloud Alibaba 的前后端分离微服务架构。第二，实现学生选课、推荐、课表、消息通知和后台管理等功能。第三，通过 Redisson 锁、请求日志和 RabbitMQ 队列减少并发选课超卖风险。第四，使用 Postman 和 JMeter 完成接口与性能测试，测试结果显示核心流程可运行，压测中未出现库存超卖。

追问应对：如果老师问“还有什么不足”，可回答当前推荐算法较轻量，主要基于专业匹配；部署以单机 Docker Compose 为主，未来可以引入 Kubernetes HPA 做弹性伸缩。

来源：PPT 第 5、34、39、40 页；论文 `gen2_part3.py:291`、`gen2_part3.py:301-308`。

## 三、架构与技术选型

### 第 7 页：为什么选择 Spring Cloud Alibaba？

推荐回答：Spring Cloud Alibaba 提供了适合 Java 微服务开发的基础设施，特别是 Nacos 服务注册与配置管理、Gateway 统一入口、服务间调用和生态集成。选课系统需要把主业务服务和选课服务分离，Nacos 可以帮助服务发现，Gateway 统一转发请求，整体符合项目的微服务目标。

追问应对：如果老师问“为什么不用 Dubbo 或纯 Spring Boot”，回答 Dubbo 更偏 RPC 服务治理，纯 Spring Boot 更适合单体应用。本系统需要网关、注册中心、跨服务调用和后续扩展能力，因此选择 Spring Cloud Alibaba 更合适。

来源：PPT 第 7 页；论文 `gen2_part3.py:301`；网关代码 `select-course-backend/new-api-gateway/src/main/resources/application.yml`；Nacos 配置 `select-course-backend/selectcourse/src/main/resources/application.yml:23`。

### 第 7-8 页：你的系统整体架构是怎样的？

推荐回答：系统采用前后端分离和微服务分层架构。前端基于 React 和 Ant Design Pro，用户请求先进入 Spring Cloud Gateway。Gateway 做统一路由、登录态校验和跨域处理，然后转发到主业务服务或选课服务。业务层包含用户、推荐、课程、选课、消息等模块，数据层包括 MySQL、Redis 和 RabbitMQ。

追问应对：如果老师问“选课服务为什么单独拆”，回答选课是并发最高、库存一致性要求最高的业务，单独拆分后可以独立限流、扩容、加锁和压测。

来源：PPT 第 7-8、24 页；论文 `gen2_part3.py:301-302`；代码 `select-course-backend/selectcourse/src/main/java/com/hardlin/selectcourse/EnrollmentController.java:85-119`。

### 第 7 页：Nacos 在系统里起什么作用？

推荐回答：Nacos 主要用于服务注册与发现。各后端服务启动后注册到 Nacos，Gateway 或其他服务可以通过服务名找到对应实例。这样后续服务副本数量变化时，不需要手动修改调用地址。

追问应对：如果老师问“现在是否真正多实例部署”，可以实事求是回答：当前毕业设计环境主要是单机部署验证，架构上保留了通过 Nacos 进行多实例发现和后续扩容的能力。

来源：PPT 第 7、40 页；论文 `gen2_part3.py:301`；代码 `select-course-backend/selectcourse/src/main/resources/application.yml:23`。

### 第 7、20 页：Gateway 做了哪些事？

推荐回答：Gateway 是系统统一入口，负责请求路由和鉴权过滤。代码中 `SaReactorFilter` 对所有路径生效，排除登录、注册和验证码等白名单接口，其余接口执行 `StpUtil.checkLogin`。此外还在网关层处理 CORS 跨域响应头和 OPTIONS 预检请求。

追问应对：如果老师问“为什么不在每个 Controller 都写登录校验”，回答统一在网关校验可以减少重复代码，保证入口规则一致；业务服务内部仍可继续扩展细粒度角色权限。

来源：PPT 第 20 页；论文 `gen2_part3.py:188`；代码 `select-course-backend/new-api-gateway/src/main/java/com/hardlin/gateway/auth/SaTokenConfig.java:27-55`。

### 第 20 页：Sa-Token 和 Redis 是怎么配合的？

推荐回答：用户登录成功后调用 Sa-Token 创建登录会话，Token 返回给前端。后续请求携带 Token，网关通过 Sa-Token 校验登录态。Redis 用于保存会话状态，使登录状态可以在分布式服务之间共享。

追问应对：如果老师问“Token 过期怎么办”，回答 Token 过期后网关校验失败，返回错误，前端根据响应跳转登录或提示重新登录。

来源：PPT 第 20 页；论文 `gen2_part3.py:184-188`；代码 `UserServiceImpl.java:107`、`SaTokenConfig.java:37-43`。

### 第 20 页：密码安全是怎么做的？

推荐回答：当前代码中注册和登录时使用固定盐值加 MD5 做密码摘要，登录时对输入密码做同样摘要后与数据库密文比对。答辩时需要说明这是毕业设计阶段的基础实现，实际生产环境建议升级为 BCrypt 或 Argon2，并使用随机盐、密码强度校验和登录失败限制。

追问应对：如果老师指出 MD5 不够安全，不要硬说安全。回答“这是当前实现的不足，论文中安全增强方向可补充升级为 BCrypt/Argon2”。

来源：PPT 第 20、40 页；代码 `select-course-backend/cloudCalculate-backend/src/main/java/com/hardlin/springbootinit/service/impl/UserServiceImpl.java:42`、`UserServiceImpl.java:69`、`UserServiceImpl.java:95`、`UserServiceImpl.java:107`。

## 四、功能与业务流程

### 第 10 页：学生端有哪些核心功能？

推荐回答：学生端核心功能包括课程查询、智能推荐、在线选课、已选课程查看、个人课表查看和消息通知。学生可以先查看推荐课程，再进入选课页面筛选课程，提交选课后通过消息中心查看成功或失败结果。

追问应对：如果老师问“推荐和选课如何衔接”，回答推荐页面可以直接发起选课请求，选课页面也可以通过课程列表选择课程，本质都调用选课接口。

来源：PPT 第 10、28-30 页；前端代码 `CourseSelectionPage.tsx:185-190`、`RecommendedCourses/index.tsx:51`。

### 第 11-12 页：管理员端负责什么？

推荐回答：管理员端主要负责课程、教师和学生基础数据维护，以及课程审核。课程只有审核通过后才进入学生可选范围，这样可以避免未确认课程被学生误选。

追问应对：如果老师问“教师能不能管理课程”，回答教师负责课程发布和维护，管理员负责更高层级的基础数据管理与审核，二者职责不同。

来源：PPT 第 11-12、31 页；前端代码 `cloudCalculate-front/myapp/src/pages/admin/Course.tsx:24-33`、`Course.tsx:134-168`；后端字段 `selectcourse/src/main/java/com/hardlin/selectcourse/entity/Curriculum.java:72`。

### 第 13 页：学生和教师端用例之间有什么关系？

推荐回答：教师发布和维护课程，管理员审核课程，审核通过后学生才能查询、推荐和选课。学生的选课行为又会影响课程剩余名额和教师查看已选学生列表，所以学生端和教师端通过课程安排与选课记录形成闭环。

追问应对：如果老师问“谁是核心用户”，回答学生是主要使用者，教师和管理员是支撑角色，共同保障选课数据准确。

来源：PPT 第 13、17 页；论文 `gen2_part2.py:49`；代码 `EnrollmentController.java:54-83`、`EnrollmentController.java:85-119`。

### 第 15 页：登录后为什么会进入推荐流程？

推荐回答：登录成功后系统可以识别当前学生身份，再根据学生专业字段查询课程并生成推荐列表。推荐依赖学生信息，所以必须在登录态明确后执行。

追问应对：如果老师问“未登录能否推荐”，回答不应允许，因为未登录无法确定学生专业和权限，也无法保证推荐结果个性化。

来源：PPT 第 15、28 页；论文 `gen2_part3b.py:27-29`；代码 `RecommendationController.java:86-114`、`CourseRecommendationService.java:44-68`。

### 第 16 页：一次完整选课请求的流程是什么？

推荐回答：学生在前端选择课程并确认后，前端提交 `studentId` 和 `curriculumId` 到选课接口。后端先检查 requestId 日志，避免重复处理；然后基于课程 ID 获取 Redisson 分布式锁；加锁成功后检查库存、检查是否已经选过该课程；校验通过后写入选课日志和选课记录，并扣减库存；最终向前端或消息模块返回处理结果。

追问应对：如果老师问“为什么不是直接写数据库”，回答直接写数据库在并发下容易产生超卖或重复写入，必须先做库存校验、幂等校验和并发控制。

来源：PPT 第 16、24 页；论文 `gen2_part2.py:49`；代码 `EnrollmentController.java:85-119`、`EnrollmentServiceImpl.java:34-45`。

### 第 16、19 页：如何防止课程超卖？

推荐回答：核心策略是“分布式锁 + 库存检查 + 选课记录写入”。代码中通过 `redissonClient.getLock("curriculum:_select_" + curriculumId)` 为每门课程建立锁，同一课程的并发选课请求串行进入库存扣减逻辑。拿到锁后再次读取课程库存，如果 `stock <= 0` 就返回课程已满，避免库存被扣成负数。

追问应对：如果老师问“数据库层有没有兜底”，回答可以进一步加数据库事务和条件更新，如 `update curriculum set stock=stock-1 where id=? and stock>0`，当前毕业设计主要通过 Redisson 锁和业务校验实现防护。

来源：PPT 第 19、24、35 页；论文 `gen2_part3.py:302`；代码 `EnrollmentController.java:96-107`、`EnrollmentController.java:127-130`。

### 第 16 页：如何防止重复选课？

推荐回答：重复选课从两层处理。第一层是 requestId 幂等日志，若同一请求已经处理过则不重复执行。第二层是查询 enrollment 表，判断该学生是否已经选过同一门课程，如果已存在就返回失败信息。

追问应对：如果老师问“requestId 和选课记录哪个更重要”，回答二者解决的问题不同。requestId 防止同一请求重复消费，选课记录防止不同请求重复选择同一门课程。

来源：PPT 第 16、24 页；论文 `gen2_part2b.py:38`；代码 `EnrollmentController.java:87-95`、`EnrollmentController.java:108-113`。

### 第 17 页：课程审核状态如何影响学生选课？

推荐回答：课程安排实体中有 `isCheck` 字段，表示课程审核状态。后台管理页显示审核状态，待审核课程需要管理员审批。学生端查询课程时只应展示审核通过且库存有效的课程，避免未审核课程进入学生可选列表。

追问应对：如果老师问“isCheck 的具体值”，回答从前端数据看 `0` 表示待审核，`1` 表示已审核通过；如果有驳回状态可扩展为 `2`。

来源：PPT 第 17、31 页；前端代码 `Course.tsx:24-33`、`Course.tsx:134-168`；后端代码 `CurriculumServiceImpl.java:35`、`Curriculum.java:72`。

### 第 19 页：课程冲突检测怎么做？

推荐回答：冲突检测不是只比较两个开始时间，而是把课程时间看成区间。系统以默认课时长度计算结束时间，并在目标课程开始和结束时间两端扩展 15 分钟缓冲，再判断另一门课的开始时间、结束时间或整体区间是否与扩展区间重叠。15 分钟缓冲是考虑学生跨教学楼转场时间。

追问应对：如果老师问“为什么是 15 分钟”，回答这是高校课间转场的经验阈值，既比简单重叠判断更贴近实际，也不会过度扩大冲突范围。

来源：PPT 第 19、37 页；论文 `gen2_part3.py:160-162`。

### 第 30 页：选课结果如何通知学生？

推荐回答：选课处理完成后系统生成通知消息，成功时提示选课成功，失败时说明课程已满、已选过该课程等具体原因。前端导航栏通过 `unReadMessage` 字段显示未读消息数量，学生可以进入消息中心查看详细内容。

追问应对：如果老师问“为什么要异步通知”，回答选课高峰下同步返回所有结果会增加请求阻塞；异步通知可以把选课处理和结果展示解耦。

来源：PPT 第 30 页；论文 `gen2_part3.py:178-181`；前端代码 `AvatarDropdown.tsx:119`、`UserMessage.tsx:273`；数据库 `cloud_calculate.sql:4171`。

## 五、推荐算法与数据设计

### 第 28 页：智能推荐算法的依据是什么？

推荐回答：当前推荐算法主要基于学生专业和课程专业标签匹配。系统先读取学生专业，再调用课程分页接口查询相关课程；专业匹配课程设置较高推荐分数 90，通用课程设置 50，最后按推荐分数降序排序，并展示推荐理由。

追问应对：如果老师问“这算不算智能推荐”，回答它是轻量级、可解释的内容匹配推荐，适合毕业设计和冷启动场景；后续可在历史选课数据积累后升级为协同过滤或混合推荐。

## 七、并发、消息队列与一致性

### 第 19、24 页：为什么要使用 RabbitMQ？

推荐回答：RabbitMQ 用于异步削峰和结果通知。选课高峰时大量请求同时进入，如果全部同步处理，会造成接口阻塞和数据库压力集中。通过消息队列可以把请求接收和后续处理解耦，消费者按能力处理消息，处理完成后再发送成功或失败通知。

追问应对：如果老师问“消息会不会丢”，回答论文设计中强调队列和消息持久化、消费者确认和幂等日志；当前答辩可以说明后续还可加入死信队列和重试告警进一步增强可靠性。

来源：PPT 第 19、24、37 页；论文 `gen2_part2b.py:36-38`、`gen2_part3.py:178`。

### 第 19 页：Redisson 锁和数据库事务有什么区别？

推荐回答：Redisson 锁用于控制多个服务实例或多个线程对同一课程库存的并发访问，重点解决分布式并发互斥问题。数据库事务用于保证一次业务操作中多条数据库写入的一致性。二者不是替代关系，高并发场景下可以组合使用。

追问应对：如果老师问“只用数据库事务行不行”，回答单机低并发可以，但多实例并发选课下仅靠普通事务可能出现竞争，需要条件更新、唯一索引或分布式锁等额外控制。

来源：PPT 第 19、24 页；代码 `EnrollmentController.java:96-119`。

### 第 16、24 页：requestId 的作用是什么？

推荐回答：requestId 是幂等标识。系统处理选课请求前先查询 enrollment_log 表，如果相同 requestId 已经存在并成功，就直接返回“已经选课成功”；如果存在但未成功，则提示稍后重试。这样可以避免网络重试或消息重复消费导致同一请求被多次执行。

追问应对：如果老师问“requestId 能不能防止不同请求重复选同一门课”，回答不能，所以还需要查询 enrollment 表判断同一学生是否已选过同一课程。

来源：PPT 第 16、24 页；论文 `gen2_part2b.py:38`；代码 `EnrollmentController.java:87-95`、`EnrollmentController.java:114-119`。

### 第 24 页：如果加锁失败怎么办？

推荐回答：代码中使用 `tryLock(500, 5000, TimeUnit.MILLISECONDS)` 尝试获取锁。如果短时间内没有拿到锁，系统返回“系统繁忙，请稍后再试”。这种设计避免请求长时间阻塞，也能在高峰期保护系统。

追问应对：如果老师问“学生体验会不会不好”，回答可以通过前端提示和重试机制优化；但在高并发保护中，快速失败比无限等待更可控。

来源：PPT 第 24 页；代码 `EnrollmentController.java:99-123`。

### 第 24 页：高并发下如何保证最终一致性？

推荐回答：系统通过库存校验、分布式锁、选课日志和消息通知共同保证。锁保证同一课程库存扣减串行化；日志保证重复请求不重复执行；选课记录保存最终业务结果；通知模块把处理结果反馈给学生。即使通知异步到达，最终数据库记录仍是判断选课成功的依据。

追问应对：如果老师问“通知失败怎么办”，回答可以补充消息重试、死信队列和补偿任务，目前论文已经把消息可靠性作为后续可增强点。

来源：PPT 第 19、24、30 页；论文 `gen2_part2b.py:36-38`、`gen2_part3.py:178-179`。

## 九、创新点与不足

### 第 37 页：你的创新点是什么？

推荐回答：主要有三点。第一，微服务架构结合 Redisson 和 RabbitMQ，形成高并发选课防护链路。第二，基于专业匹配的轻量级可解释推荐，为学生提供推荐理由。第三，课程冲突检测引入 15 分钟转场缓冲，比简单时间相等判断更贴近实际教学场景。

追问应对：如果老师问“创新程度够不够”，回答创新点偏工程实践创新，不是提出全新算法，而是把成熟技术组合到高校选课业务中解决实际问题。

来源：PPT 第 37 页；论文 `gen2_part2.py:16`、`gen2_part3.py:302`。

### 第 40 页：系统目前有哪些不足？

推荐回答：第一，推荐算法目前主要基于专业字段，维度较少。第二，部署环境以单机 Docker Compose 为主，还没有真正上 Kubernetes 集群。第三，安全上当前密码摘要实现较基础，生产环境应升级密码哈希和权限控制。第四，移动端体验还可以继续扩展。

追问应对：如果老师问“哪个最优先改”，回答优先改安全和高并发部署，其次提升推荐算法，因为上线系统首先要保证安全性和稳定性。

来源：PPT 第 40 页；论文 `gen2_part3.py:307-312`；代码 `UserServiceImpl.java:42`、`UserServiceImpl.java:95`。

### 第 40 页：为什么未来考虑 Kubernetes？

推荐回答：当前系统已按微服务拆分，但主要运行在单机环境。Kubernetes 可以通过 Deployment 管理服务副本，通过 Service 做集群内部负载均衡，通过 HPA 根据 CPU 或请求指标自动扩缩容，更适合选课高峰这种短时流量激增场景。

追问应对：如果老师问“你现在有没有实现 Kubernetes”，回答当前论文以设计和未来扩展为主，实际开发阶段主要使用 Docker Compose 或本地环境完成系统实现和测试。

### 老师问：论文题目里“智能”体现在哪里？

### 老师问：论文题目里“微服务”体现在哪里？

推荐回答：系统不是单个后端应用完成所有逻辑，而是有 Gateway 统一入口、主业务服务和选课微服务。选课高并发逻辑位于独立的 selectcourse 服务中，推荐服务通过跨服务调用获取课程数据，Nacos/Gateway 支撑服务发现和路由。

推荐回答：Redis 在系统中承担会话共享和分布式锁支撑。如果 Redis 不可用，会影响登录态校验和高并发选课锁。生产环境应使用 Redis 哨兵或集群提升可用性，并为关键接口设置降级策略。当前毕业设计阶段重点验证 Redis 正常情况下的并发控制能力。

### 老师问：如果 RabbitMQ 消费失败怎么办？

推荐回答：论文设计中强调消息持久化、消费者确认和 requestId 幂等日志。消费失败时可以重回队列或重试；实际生产环境还应加入死信队列、最大重试次数和告警机制，避免消息无限重试。

### 老师问：你如何证明没有超卖？

推荐回答：从实现上，同一课程通过课程 ID 建立 Redisson 锁，进入临界区后再次检查库存，库存不足直接失败。从测试上，JMeter 对选课接口进行并发压测，统计表中库存超卖次数为 0。实现和测试相互印证。

来源：PPT 第 19、24、35 页；论文 `gen2_part3.py:245-255`；代码 `EnrollmentController.java:96-107`。

推荐回答：前端判断只是减少用户无效点击，不能作为安全依据。用户可以绕过前端直接调用接口，或者在并发场景下前端库存已经过期，所以后端必须重新读取库存并在锁内判断。

## 十一、答辩回答策略

1. 回答架构问题时，先说业务压力，再说技术选择，不要直接堆技术名词。

2. 回答代码实现时，优先引用三条主线：Gateway 鉴权、Redisson 选课锁、推荐服务评分。

3. 回答测试问题时，必须说出 Postman 和 JMeter 的分工，并强调指标包括平均响应时间、吞吐量、错误率、库存超卖次数。

4. 回答不足时，主动承认 MD5、单机部署、推荐算法简单这些点，并给出明确改进方案。

5. 如果老师问到 PPT 某页图，按页码回答：第 12 页讲管理员基础数据维护，第 13 页讲学生/教师业务闭环，第 24 页讲高并发选课链路，第 35 页讲测试证据。
