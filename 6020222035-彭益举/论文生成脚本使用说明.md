# 论文生成脚本使用说明

## 1. 目标

本目录下的脚本用于生成毕业论文成稿 `docx/pdf`，并统一维护以下内容：

- 封面、摘要、目录、页眉页脚、分页
- 第 1 章到第 7 章正文
- 图表插入
- 参考文献、致谢
- Linux 环境下的静态目录回写

当前维护原则：

- 论文正文不再插入大段源码
- 论文描述应以实现流程说明为主
- 论文内容应尽量与项目真实代码一致

## 2. 目录结构

本目录中与论文生成直接相关的核心文件如下：

- `generate_paper.py`
  兼容入口。建议平时直接运行这个文件。
- `gen2_main.py`
  当前主入口，负责加载各分段脚本、生成 `docx`、做静态目录回写。
- `gen2_part1.py`
  负责格式基础设施、封面、摘要、目录、样式定义。
- `gen2_part2.py`
  负责第 1 章到第 3 章。
- `gen2_part3.py`
  负责第 4 章到第 7 章、参考文献、致谢。
- `figures/`
  论文插图目录。正文中的截图、图示都从这里读取。
- `2425_41_10475_080902_6020222035_LW.docx`
  当前生成的论文成稿。
- `2425_41_10475_080902_6020222035_LW.pdf`
  当前生成的 PDF 版本。

## 3. 推荐操作方式

### 3.1 生成论文

在本目录执行：

```bash
python3 generate_paper.py
```

生成完成后，默认输出：

- `2425_41_10475_080902_6020222035_LW.docx`

如果系统安装了 LibreOffice，也可以继续转 PDF：

```bash
libreoffice --headless --convert-to pdf --outdir . 2425_41_10475_080902_6020222035_LW.docx
```

### 3.2 推荐修改流程

推荐按照下面顺序修改：

1. 先改脚本，不直接手改最终 `docx`
2. 运行 `python3 generate_paper.py`
3. 打开生成后的 `docx/pdf` 检查效果
4. 确认无误后再提交 Git

原因：

- 直接手改 `docx` 无法沉淀成可重复生成的结果
- 后续再生成时，手工改动会被覆盖

## 4. 各文件负责什么

### 4.1 `gen2_part1.py`

适合放这些内容：

- 页面样式
- 字号、行距、标题样式
- 封面
- 摘要、英文摘要
- 目录生成
- 图题、表题、参考文献格式

如果你要改：

- 页边距
- 页眉页脚
- 目录
- 正文字体字号
- 图题/表题样式

优先改这个文件。

### 4.2 `gen2_part2.py`

适合放：

- 第 1 章 绪论
- 第 2 章 技术分析
- 第 3 章 可行性与需求分析

如果你要改：

- 研究背景
- 国内外研究现状
- 技术栈说明
- 需求分析

优先改这个文件。

### 4.3 `gen2_part3.py`

适合放：

- 第 4 章 概要设计
- 第 5 章 详细设计与实现
- 第 6 章 系统测试
- 第 7 章 总结与展望
- 参考文献
- 致谢

如果你要改：

- 图表对应关系
- 测试表格
- 各模块实现说明
- 参考文献
- 致谢

优先改这个文件。

## 5. 图片怎么换

论文中的图片统一从 `figures/` 读取。

### 5.1 换图步骤

1. 先把新图片放到 `figures/`
2. 在 `gen2_part3.py` 或 `gen2_part2.py` 中找到对应 `insert_fig(...)`
3. 替换图片文件名、图题、宽度
4. 重新运行 `python3 generate_paper.py`

示例：

```python
insert_fig('fig_09_p38.png', '图 5.2  选课页面', 14)
```

含义：

- 第 1 个参数：图片文件名
- 第 2 个参数：图题
- 第 3 个参数：图片宽度，单位约为厘米

### 5.2 图片修改注意事项

- 图题必须和图片内容一致
- 正文里的描述必须和图题一致
- 尽量统一截图风格，不要一张深色一张浅色
- 不要把有学校错名、个人站点信息、水印、无关 UI 的截图放进论文

## 6. 目录与分页说明

当前脚本已经处理了 Linux 下目录不自动展开的问题。

### 6.1 当前策略

- 先生成 `docx`
- 再通过 LibreOffice 计算目录页码
- 把目录静态回写到文档中

这意味着：

- Linux 下打开论文时，不会只看到“目录将随文档打开或字段更新后自动生成”这类占位提示
- 不依赖 Word/WPS 手动更新目录字段

### 6.2 相关环境要求

需要本机安装：

- `python3`
- `python-docx`
- `libreoffice`

如果 LibreOffice 异常，通常会影响：

- 目录页码提取
- PDF 导出

## 7. 修改正文时的注意事项

### 7.1 不要写进论文的内容

当前已明确不建议写入：

- 大段 Java / TypeScript / 配置代码
- 成片注解、方法签名、泛型、调用链
- 过度实现细节，例如精确到每个方法名、每行 API 名称

建议写法：

- 说明模块职责
- 说明处理流程
- 说明关键判断逻辑
- 说明数据流和控制流

### 7.2 事实一致性要求

修改论文时，要优先和真实项目代码对齐，尤其注意：

- Spring Boot 版本
- Java 版本
- 鉴权方式
- 密码校验方式
- Token 有效期
- 通知内容
- 冲突检测逻辑
- 测试数据与测试结论

如果论文与代码不一致，优先改论文，不要先凭印象改描述。

### 7.3 图文一致性要求

任何截图都要满足：

- 图题和图片一致
- 正文描述和图片一致
- 页码、章节号、图号连续

## 8. 参考文献修改注意事项

- 优先使用中文文献
- 优先保留已核实过的条目
- 不要保留与本文无明显关系的技术书籍或论文
- 修改参考文献后，要同步检查正文引用语境是否还合理

## 9. Git 使用建议

推荐每完成一个清晰阶段就提交一次，例如：

- 修完格式
- 修完图文对应
- 修完逻辑一致性
- 去掉代码块
- 更新参考文献

推荐命名方式：

- `docs(thesis): fix figure-text alignment`
- `docs(thesis): align thesis wording with implementation`
- `docs(thesis): remove embedded code excerpts`

## 10. 当前维护建议

后续继续维护时，建议优先检查这几类问题：

1. 图题、图片、正文是否一致
2. 表格是否过宽、跨页是否难看
3. 标题是否会出现在页末孤行
4. 目录页码是否与正文一致
5. 摘要、正文、结论中的技术口径是否一致

## 11. 一条最重要的规则

不要直接把最终 `docx` 当作主编辑对象。

主编辑对象应始终是：

- `gen2_part1.py`
- `gen2_part2.py`
- `gen2_part3.py`
- `gen2_main.py`

最终 `docx/pdf` 只是脚本生成结果。
